7 AmpGen is a library and set of applications for fitting and generating multi-body particle decays using the isobar model.
8 It developed out of the MINT project used in the fitting of three and four-body pseudoscalar decays by the CLEO-c and LHCb colloborations. The library can handle the fitting and generation of a wide variety of final states, including those involving fermions and photons, as well as polarised initial states.
9 
10 Source code for the evaluation of amplitudes is dynamically generated by a custom engine, JIT compiled and dynamically linked to the user programme at runtime, which results in high flexibility and performance.

20 ## Getting started
21 ### Installation
22 #### Getting the source
23 Clone with git
24 ```
25 git clone http://github.com/GooFit/AmpGen/ --recursive
26 ```
27 There is at the time of writing only a master branch (FIXME)

29 ##### Build requirements:
30 * cmake >= 3.11.0
31 * C++ compiler with CXX standard >= 14 (gcc >= 4.9.3, clang ~ 5).
32  Defaults to Cxx17 (enable cxx14 with cmake flag `-DCMAKE_CXX_STANDARD=14` )
33 * ROOT >= 6 with MathMore
34  To (re)configure root with these options, use the additional command line options `-Dcxx14 -Dmathmore=ON` when configuring the installation of ROOT.
35 
36 ##### Optional:
37 * boost >= 1.67.0 for unit tests
38 * xROOTd for network file access
39 * OpenMP for multithreading
40 * ROOT >= 6 with MathMore and Minuit2 enabled. The external version of Minuit2 provided as an external package of GooFit is used if the ROOT version is not unavailable.
41  To (re)configure root with these options, use the additional command line options `-Dcxx14 -Dminuit2=ON -Dmathmore=ON` when configuring the installation of ROOT.
42 
43 #### Building
44 The configuration of the AmpGen build is performed by cmake.
45 It is recommended to use a build directory to keep the source tree clean.
46 
47 ```shell
48 mkdir build
49 cd build
50 cmake ..
51 make
52 ```
53 This will build the shared library, several standalone test applications, and a set of unit tests.

55 #### Usage with ROOT
56 
57 The library can be used interactively in conjunction with the ROOT C++ interpreter by adding the following lines
58 to the users root login script
59 
60 ```
61  gSystem->Load("path_to_ampgen/build/lib/libAmpGen.so");
62  gROOT->ProcessLine(".include path_to_ampgen");
63 ```
64 ##### LLVM
65 You can also build AmpGen with LLVM. The only change you might want when using Apple LLVM
66 is to specifically specify the location of the build tool for AmpGen's JIT:
67 
68 ```shell
69 -DAMPGEN_CXX=$(which c++)
70 ```

81 Several examples of usages of the library are included in the apps directory and are

87 ### Options files and decay descriptors
88 
89 Options files will generally contain the description of one or more particle decays,
90 as well as other settings such as input/output locations, global flags such as
91 whether complex numbers should be interpreted as cartesian or polar, and other parameters
92 such as the masses and widths of particles is these differ from those given in the PDG.
93 
94 A minimal example options file for the generator application could contain:
95 ```
96 EventType D0 K+ pi- pi- pi+
97 # Real / Amplitude | Imaginary / Phase
98 # Fix? Value Step | Fix? Value Step
99 D0{K*(892)0{K+,pi-},rho(770)0{pi+,pi-}} 2 1 0 2 0 0
100 ```
101 The EventType specifies the initial and final states requested by the user.
102 This gives the ordering of particles used in input data, in output code, and used in internal computations.
103 This also defines how the amplitude source code must be interfaced with external packages, i.e. MC generators such as EvtGen.
104 
105 The decay products of a particle are enclosed within curly braces, for example
106 ```
107 K*(892)0{K+,pi-}
108 ```
109 describes an excited vector kaon decaying into a charged kaon and pion.
110 For more details about the API for describing particle decays, see [AmpGen::Particle](https://goofit.github.io/AmpGen/de/dd7/class_amp_gen_1_1_particle.html)
111 The other numbers on the lines that describe the decays parameterise the coupling to this channel,
112 either in terms of real and imaginary parts or an amplitude and a phase.
113 Each parameter is specified in terms of three numbers: the _fix_ flag, the initial value, and the step size.
114 The possible options for the _fix_ flag are:
115 * Free (fix=0) and a step size of not 0.
116 * Fixed (fix=2, for historical reasons)
117 * Compile-Time-Constant (fix=3) which indicates that the parameter should be treated as a (JIT) compile time constant, which in some cases allows for more aggressive optimisations to be performed.
118 
119 These options can be used in the Generator application, which is described below.
120 
121 Decays can either be specified fully inline, as above, or split into multiple steps, which is useful for treating the so called _cascade_ decays, an example of which is shown below.
122 ```
123 D0{K(1)(1270)+,pi-} 0 1 0.1 0 0 0.1
124 
125 K(1)(1270)+{rho(770)0{pi+,pi-},K+} 2 1 0 2 0 0
126 K(1)(1270)+{K*(892)0{K+,pi-},pi+} 0 1 0.1 0 0 0.1
127 ```
128 The production/decay couplings of the @f$K_1(1270)^+@f$ resonance are now defined in terms of the coupling to the @f$\rho(770)^0 K^+@f$ channel,
129 which can be useful in making comparisons between different production modes of a resonance.
130 Additional care must be taken in such a case to not introduce redundant degrees of freedom.
131 
132 Configuration can be split over multiple files by the using _Import_ keyword, for example, to import the parameters for the isoscalar K-matrix, the line
133 ```
134 Import $AMPGENROOT/options/kMatrix.opt
135 ```
136 
137 can be added to options file. Multiple user configuration files can also be specified by including multiple files on the command line.

148 ### Generator
149 
150 The standalone generator for models can be used as
151 
152 ```shell
153 Generator MyOpts.opt --nEvents=10000 --Output=output.root
154 ```
155 
156 Which generates 10000 events of the model described in MyOpts.opt and saves them to output.root.
157 The output should include a tree (DalitzEventList) of candidates with the full four-vectors, as well as one- and two-dimensional projections, an example of which is shown below:
158 
159 <p align="center">
160  <img src="https://github.com/GooFit/AmpGen/raw/master/doc/figs/s01.png">
161 </p>
162 
163 In particular, the tree indicates the way in which data is by default loaded into the _Event_ / _EventList_ class.
164 Several models for different @f$D^0@f$ decays published by the LHCb collaboration are included in the options directory. These options do not include EventTypes so as they can be included as part of a larger decay chain if required. For example, to generate @f$10^6@f$ toy @f$D^{0}\to K^{-} \pi^+ \pi^+ \pi^- @f$ events:
165 
166 ```
167 ./Generator options/D02Kpipipi.opt --EventType "D0 K- pi+ pi+ pi-" --nEvents 1000000
168 ```

170 ### Debugger
171 
172 The debugger application is a tool for producing verbose information debugging for amplitudes and models. It is used on an options file as
173 ```
174 ./Debugger MyOpts.opt
175 ```
176 which calculates each amplitude at a randomly generated point in phase space, as well as calculating the total amplitude accounting for complex
177 couplings. Also computed is the amplitude for the P conjugate event. A more useful application is the _verbose_ debugging mode which can be
178 activated by
179 ```
180 ./Debugger MyOpts.opt --CoherentSum::Debug
181 ```
182 which produces a large number of intermediate steps in the calculation of each amplitude, which are added to the calculation using the
183 ADD_DEBUG and ADD_DEBUG_TENSOR macros in the code generation. For example, in src/Lineshapes/BW.cpp.
184 (n.b), if the model is a PolarisedSum, i.e. handles spin in the initial/final state, the flag PolarisedSum::Debug should be used instead of CoherentSum::Debug, in order
185 to generate all of the intermediate calculations of the amplitude.

187 ### ConvertToSourceCode
188 
189 This produces source code to evaluate the PDF, and normalises for use with other generators such as EvtGen, i.e. P(max) < 1. This can be used as
190 ```shell
191 ./ConvertToSourceCode MyOpts.opt --Output=MyFile.cpp
192 ```
193 This can then be a compiled to a shared library using
194 ```shell
195 g++ -Ofast -shared -rdynamic --std=c++14 -fPIC MyFile.cpp -o MyFile.so
196 ```

200 ### SignalOnlyFitter
201 
202 An example fitter is provided in _examples/SignalOnlyFitter.cpp_, which as the name suggests only has a single signal component in the fit.
203 The free parameters of the fit are specified in the same way as the Generator,
204 with the additional relevant slots being _DataSample_ which specifies the signal sample to fit,
205 which is presumed to already have the selection applied, and _Branches_ which takes a list of branch names,
206 and defaults to the format used by the Generator etc. More details can be found with
207 ```
208 SignalOnlyFitter --help
209 ```
210 For example, the fitter can be used to fit a toy MC sample generated by the generator by running:
211 ```
212 Generator MyOpts.opt --nEvents 100000
213 SignalOnlyFitter MyOpts.opt --DataSample Generate_Output.root
214 ```

227 ### Python Bindings
228 Models built into a shared library can be used in python using the following flags into ConvertToSourceCode:
229 ```shell
230 ./ConvertToSourceCode MyOpts.opt --Output=MyFile.cpp --OutputEvents=events.csv --IncludePythonBindings
231 ```
232 where normalisation events are also output for testing purposes in events.csv.
233 This can then be used with the python bindings in ampgen.py:
234 ```python
235 from ampgen import FixedLib
236 model = FixedLib('MyModel.so')
237 print(model.matrix_elements[0]) # Print first matrix element
238 
239 import pandas as pd
240 data = pd.read_csv('events.csv', nrows=100_000)
241 fcn1 = model.FCN_all(data)
242 
243 # or, a bit slower, but just to show flexibility:
244 fcn2 = data.apply(model.FCN, axis=1)
245 ```

247 ### Particle Properties and Lineshape parameters
248 The particles available and their default properties can be found in *options/mass\_width.csv* using the MC format of the 2008 PDG. Additional pseudoparticles, such as nonresonant states and terms for use in conjunction with K-matrices are defined by *options/MintDalitzSpecialParticles.csv*. Any additional user defined particles should be added here. For the default lineshape (the relavistic Breit-Wigner or BW), there are three parameters: The mass, the width and the Blatt-Weisskopf radius. These default to their PDG values, but can be overridden in the options file with parameters: *particleName*\_mass, *particleName*\_width, *particleName*\_radius. To vary the mass of the @f$K_1(1270)^+@f$ meson, the line:
249 ```
250 K(1)(1270)+_mass 0 1.270 0.01
251 ```
252 could be added to the user configuration.
253 Other lineshapes may define other parameters, for example channel couplings or pole masses in the case of the K-matrices, can be set or varied in a similar manner.
254 
255 ### Fit parameters and expressions
256 
257 Parameters can either be specified by three parameters, in the case of a scalar parameter such as a mass or a width, or with six parameters in the case of a complex parameter such as a coupling.
258 Upper and lower bounds on parameters can also be set by specifying a parameter with five parameters or ten parameters for a scalar or complex, respectively.
259 For example, if we wished to vary the mass of the @f$K_1(1270)^+@f$ meson in the above example, but restricting the allowed values in the range @f$[0.0,2.0]\,\mathrm{GeV}@f$:
260 ```
261 K(1)(1270)+_mass 0 1.27 0.01 0.0 2.0
262 ```
263 
264 Parameters can also be related to each other via expressions,
265 Suppose for example we have @f$K_1(1270)^+@f$ and @f$K_1(1270)^-@f$ in the same fit (for example, in @f$D^0 \to K^- K^+ \pi^- \pi^+@f$)
266 The properties of one can be allowed to vary, for example the @f$K_1(1270)^+@f$, and the other fixed to the same value, using:
267 ```
268 K(1)(1270)+_mass 0 1.27 0.01 0.0 2.0
269 K(1)(1270)bar-_mass = K(1)(1270)+_mass
270 ```
271 Parameter expressions are whitespace delimited due to the abundance of **odd** glyphs such as brackets and +/- in the names of parameters.
272 Expressions support the binary operations @f$(+,-,/,* )@f$, as well as common unary functions such as sqrt, trigonometric functions etc.

274 ### Spin Formalisms
275 
276 AmpGen implements both the covariant tensor (or Rarita-Schwinger) and canonical helicity formalism for describing the angular momentum component of decays.
277 Both formalisms refer to states of well-defined orbital angular momentum, as opposed to the helicity states, as the states with well-defined orbital angular momentum have a straightforward parity and momentum dependences.
278 The default formalism is the covariant tensor formalism, but this can be switched to the canonical formalism changing the flag
279 ```
280 Particle::SpinFormalism Canonical ## default = Covariant
281 ```
282 in the options file.
283 The spin formalism for an individual decay chain can be specified by changing the attribute SpinFormalism in the decay descriptor. For example,
284 ```
285 D0[SpinFormalism=Canonical]{K*(892)bar0,rho(770)0}
286 ```
287 selects the S-wave of the @f$K^* \rho@f$ system. The user can also specify systems of helicity couplings in the canonical formalism, using the attribute _helAmp_. For example, suppose the transversity amplitudes were used rather than the canonical, then the user can specify
288 ```
289 D0[SpinFormalism=Canonical;helAmp=Long]{K*(892)bar0,rho(770)0}
290 D0[SpinFormalism=Canonical;helAmp=t1]{K*(892)bar0,rho(770)0}
291 D0[SpinFormalism=Canonical;helAmp=t2]{K*(892)bar0,rho(770)0}
292 ```
293 For the longitudinal and two transverse amplitudes. These must then be defined by the user in terms of the helicity amplitudes in the following structure:
294 ```
295 Long {
296  1.0 0 0
297 }
298 
299 t1 {
300  0.707106781 +1 +1
301  0.707106781 -1 -1
302 }
303 
304 t2 {
305  0.707106781 +1 +1
306  -0.707106781 -1 -1
307 }
308 ```
309 That is specified as sets of three numbers, firstly the coupling, and then the two particle helicities. So in this example, the longitudinal amplitude is the $00$ helicity state, while the two transverse amplitudes and the sum and difference of the two other helicity amplitudes.

311 ### Quasiparticles
312 
313 Quasiparticles are (semi)fictional decaying particles that can be implemented in the decay chain for a variety of different purposes. The original use case was to group some amplitudes with the same quantum numbers with couplings that want to be factorised.
314 For example, for the @f$I=0@f$ @f$S@f$-wave, the K matrix description may be used. The coupling from the initial state may be written as
315 ```
316 D0{K0S0,NonResS0} ...
317 ```
318 The quasiparticle _NonResS0_ can then be decayed to the final state via the K matrix lineshape(s)
319 ```
320 NonResS0[kMatrix.pole.0]{pi+,pi-} ...
321 NonResS0[kMatrix.pole.1]{pi+,pi-} ...
322 ...
323 ```
324 where each of the .X is one of the terms of the (pole) production vector.
325 In this example, an equivalent formulation would be
326 ```
327 D0{K0S0,NonResS0[kMatrix.pole.0]{pi+,pi-}} ...
328 D0{K0S0,NonResS0[kMatrix.pole.1]{pi+,pi-}} ...
329 ...
330 ```
